## 1. 简历编辑器核心

### 1.1 简历内容编辑（P0）

**描述**：用户可以在编辑器中创建和编辑简历内容。编辑器是产品的核心交互面。

**边界**：
- 范围内：简历各模块（个人信息、教育经历、工作经历、技能、项目经验等）的创建、编辑、排序
- 已裁决：Markdown 语法支持（用户在 textarea 中输入 markdown，Typst 模板解析渲染；不引入富文本编辑器库）
- 不在范围内：协同编辑（intent.md 未提及多人协作）
- 不在范围内：简历之外的文档类型（如求职信、Portfolio）

**验收标准**：
- 用户可以新建一份空白简历
- 用户可以添加、编辑、删除简历中的每个模块
- 用户可以调整模块顺序
- 编辑内容实时反映到预览中
- 编辑操作支持撤销/重做

### 1.2 简历持久化（P0）

**描述**：用户编辑的简历数据需要持久化存储，关闭浏览器后再次访问不丢失。

**边界**：
- 范围内：本地持久化（浏览器端存储）
- 已裁决：纯浏览器本地存储（IndexedDB），无云端账户系统

**验收标准**：
- 用户关闭浏览器后重新打开，简历数据完整保留
- 用户可以管理（查看列表、删除）已保存的简历

## 2. Typst 渲染系统

### 2.1 简历渲染（P0）

**描述**：使用 Typst 作为简历排版引擎，将用户编辑的简历内容渲染为可视化输出。

**边界**：
- 范围内：Typst 渲染简历为可预览的视觉输出
- 范围内：实时或接近实时的渲染预览
- 不在范围内：用户直接编辑 Typst 源码（用户操作的是结构化编辑器，非代码编辑器）

**验收标准**：
- 用户编辑内容后，预览区域在合理时间内（≤2 秒）更新渲染结果
- 渲染结果排版正确，无内容截断或溢出
- 中文内容渲染正常，无乱码或字体缺失

### 2.2 多模板系统（P0）

**描述**：提供多种简历模板供用户选择，模板控制简历的视觉布局和排版风格。

**边界**：
- 范围内：预置多种模板，用户可切换
- 范围内：模板决定布局和视觉风格，内容与模板分离
- 已裁决：初始版本 5 个模板（经典单栏、双栏、学术风、现代、极简）
- 已裁决：模板只读，用户不可自定义

**验收标准**：
- 提供 5 个预置模板（classic、twocolumn、academic、modern、minimal）
- 用户切换模板时，内容自动适配新模板布局
- 模板切换不丢失任何用户已编辑的内容
- 每个模板的排版在 A4 纸面尺寸下正确输出

### 2.3 简历导出（P1）

**描述**：用户可以将渲染好的简历导出为文件。

**边界**：
- 范围内：导出为 PDF（简历的标准交付格式）
- 已裁决：仅 PDF，不支持 DOCX / PNG

**验收标准**：
- 用户可以一键导出当前简历为 PDF 文件
- 导出的 PDF 与预览视觉一致
- PDF 中的文字可选中、可搜索（非图片化 PDF）

## 3. AI 智能优化

### 3.1 AI 优化选项（P0）

**描述**：提供若干预设的 AI 优化选项，每个选项绑定对应的提示词结构，降低用户使用 AI 的认知门槛。用户不需要理解提示词工程，只需选择优化方向。

**边界**：
- 范围内：预设 AI 优化选项（如"让表述更专业"、"量化工作成果"、"精简冗余内容"等）
- 范围内：每个选项对应一套提示词模板
- 范围内：用户选择后，AI 基于当前简历内容和选中选项执行优化
- 范围内：用户可以预览 AI 修改建议，选择接受或拒绝
- 已裁决：最小集 3-4 个预设选项（润色表述、量化成果、精简内容、匹配岗位）
- 已裁决：AI 增加"从零生成"模式 — 用户描述职业背景，AI 生成简历初稿（解决冷启动死胡同）
- 已裁决：聊天式交互 — AI 面板维护对话历史上下文，每次调用携带前几轮对话，支持多轮对话
- 已裁决：自由文本输入走 AI 自动路由 — 轻量 AI 调用根据用户输入自动选择最合适的 system prompt，而非硬编码到某个预设选项

**验收标准**：
- 用户可以看到清晰的 AI 优化选项列表，每个选项有直观的名称和简短描述
- 选择一个选项后，AI 在合理时间内返回优化建议
- 用户可以对比优化前后的差异
- 用户可以逐条接受或拒绝 AI 的修改建议
- AI 优化不修改用户未选中的内容区域
- **（3.3 依赖）** AI 的输入和输出均基于结构化 JSON，而非纯文本拼接（详见 3.3）
- **（3.3 依赖）** 用户的自由文本指令（输入框、suggestion chips 的原始文本）必须作为 user message 的一部分传给 AI，不得仅用于关键词匹配

### 3.2 多 AI 供应商配置（P1）

**描述**：支持多种 AI API 提供商，初始集成 OpenRouter，架构上预留多供应商切换能力。

**边界**：
- 范围内：OpenRouter 作为首个集成的 AI 供应商
- 范围内：架构支持未来接入其他供应商（如直连 OpenAI、Anthropic 等）
- 范围内：用户可以配置自己的 API Key
- 已裁决：不提供官方 Key，用户自行配置 API Key

**验收标准**：
- 通过 OpenRouter 成功调用 AI 服务完成简历优化
- AI 服务调用失败时，给出清晰的错误提示（如 API Key 无效、余额不足、网络超时）
- API Key 配置入口清晰可达，操作步骤不超过 3 步

### 3.3 AI 上下文工程 -- JSON 中间数据层（P0）

**描述**：Resume JSON 作为用户与 AI 之间的共享数据层。用户通过 UI 编辑 JSON，AI 通过 prompt 读取 JSON 并返回修改后的 JSON，系统将 AI 返回的 JSON 合并回 Resume 数据。这解决当前三个已确认的系统性缺陷。

**当前问题（驱动本需求的背景）**：

1. **上下文序列化丢失结构**：当前将简历各 section 的 description 字段用 `\n\n` 拼接成纯文本传给 AI。AI 无法区分教育经历与工作经历，看不到字段结构（公司名、职位、时间等独立字段），丢失技能 level 信息和 section 排序/可见性。
2. **结果写回错位**：AI 返回的纯文本一律覆写到 `personal.summary`，无论用户请求优化哪个模块。其他模块的修改无法落地。
3. **用户意图丢失**：输入框和 suggestion chips 的文本仅用于关键词匹配选择 optionId，用户的具体指令（如"用 STAR 法则重写工作经历"）没有传给 AI。

**边界**：

- 范围内：将 Resume 数据以结构化 JSON 格式序列化到 AI prompt 的 user message 中
- 范围内：AI 返回结构化 JSON，系统解析后合并回 Resume 数据模型
- 范围内：用户的自由文本指令作为 AI prompt 的一部分传递，不仅用于选择预设选项
- 范围内：Section 级别的定向优化（用户可选择只优化某个模块，AI 仅返回该模块的修改）
- 范围内：AI 返回内容的 JSON Schema 校验，防止脏数据写入
- 范围内：AI 返回 JSON 解析失败时的容错处理和用户提示
- 不在范围内：AI 修改简历的元数据字段（id、createdAt、updatedAt、templateId 等系统字段）
- 范围内：AI 可在现有 section 内增删 item（如添加一条工作经历），系统为新增 item 自动生成 id
- 不在范围内：AI 增删 section 类型（如自动添加"荣誉奖项"模块）

**子需求 3.3.1：上下文序列化（Resume -> AI）**

将 Resume 数据模型序列化为 AI 可理解的 JSON 结构。序列化结果必须包含：
- 每个 section 的类型标识（education / work / skills / projects / custom）
- 每个 item 的全部结构化字段（如 WorkItem 的 company、position、startDate、endDate、description）
- 技能的 level 信息
- section 的排序和可见性状态
- 个人信息（personal）的全部字段

序列化结果必须排除：
- 系统字段（id、createdAt、updatedAt、templateId）
- item 级别 id（AI 不需要 id，写回时按 section 类型 + 数组整段替换）

**子需求 3.3.2：指令传递（User Intent -> AI）**

用户意图必须完整传递给 AI，包括：
- 预设选项的 system prompt（润色/量化/精简/匹配岗位）
- 用户在输入框中的自由文本指令（原文传递，不做关键词截断）；"匹配岗位"场景下用户在此输入框描述目标岗位，无需专门的 JD 粘贴区
- 当用户点击某个 section 的"墨灵"按钮时，需携带目标 section 标识，指示 AI 聚焦该 section
- prompt 结构：system message（预设 prompt）+ user message（整份简历 JSON + 目标 section 标识 + 用户自由文本指令）
- AI 返回要求：仅返回目标 section 的 JSON（不返回整份简历），减少 token 消耗和写回复杂度

**子需求 3.3.3：结果写回（AI -> Resume）**

AI 返回的 JSON 必须经过以下流程才能写入 Resume：
- JSON 语法解析（处理 AI 可能返回的 markdown 代码块包裹、多余文本等情况）
- 结构校验（返回的 JSON 是否符合 Resume 数据模型的子集）
- 按 section 类型整段替换（AI 返回的目标 section 数组直接替换原 section 数组，不做字段级别合并）
- 数组长度变化支持：AI 可以新增或删除 item（如添加一条工作经历、删除冗余条目），新增 item 由系统自动生成 id
- 系统字段保护（id、createdAt、updatedAt、templateId 不可被 AI 返回值覆盖；保留的 item 保持原 id）
- 写回前展示前后对比视图，用户一次性接受或拒绝全部修改

**验收标准**：

- AC1：AI 收到的 prompt 中包含完整的简历 JSON 结构，不是纯文本拼接
- AC2：AI 能区分不同 section 类型，对"润色工作经历"和"润色教育经历"的请求分别作用于正确的数据
- AC3：AI 返回的修改精准写回对应字段，不影响其他 section 或字段
- AC4：用户在输入框中的自由文本指令被完整传递给 AI（可通过浏览器 Network 面板验证请求体）
- AC5：AI 返回格式异常（非法 JSON、结构不匹配）时，系统不崩溃，向用户展示有意义的错误提示，原始简历数据不受影响
- AC6：section 级别的"墨灵"按钮触发时，AI 仅修改目标 section 的内容
- AC7：系统字段（id、createdAt 等）在任何 AI 交互后保持不变

**优先级**：P0 -- 此需求是 3.1（AI 优化选项）正确工作的前提。当前实现中 3.1 的"AI 优化不修改用户未选中的内容区域"这条验收标准事实上无法通过。

## 4. 中古风设计系统

### 4.1 古风视觉体系（P0）

**描述**：整个应用的视觉语言以中国古典文人文化为基准，核心元素包括宣纸、墨砚、印章、窗棂、祥云。这些元素不是纯装饰，每个都有功能对应。

**边界**：
- 范围内：古风设计令牌体系（色彩、字体、间距、圆角等）
- 范围内：核心古风元素的功能映射（宣纸=内容背景、印章=确认操作、窗棂=导航/分割、祥云=装饰/空态、墨砚=AI 相关功能区域）
- 范围内：中国传统色系作为色彩基础
- 不在范围内：过度拟物化（不是仿古软件，是有古风气韵的现代应用）

**验收标准**：
- 应用整体视觉一致，古风元素融入自然，不突兀
- 色彩体系以 DESIGN.md 中定义的中国传统色为基础
- 所有功能性古风元素（宣纸背景、印章按钮、窗棂导航等）可识别且可交互
- 在不同屏幕尺寸下古风元素不失真、不错位

### 4.2 原子化设计组件（P0）

**描述**：采用原子化设计方法论（Atoms → Molecules → Organisms → Templates → Pages），构建可复用的组件体系。

**边界**：
- 范围内：基于古风设计令牌的原子组件（按钮、输入框、卡片、分割线等）
- 范围内：原子组件组合为分子/有机体级别的复合组件
- 范围内：组件文档或 Storybook 式的组件展示
- 不在范围内：向外部发布组件库（这是产品内部的设计系统）

**验收标准**：
- 所有 UI 元素由原子组件组合而成，无内联样式或一次性组件
- 设计令牌变更后，所有引用该令牌的组件自动更新
- 每个原子组件有明确的 props 接口和使用约束

### 4.3 动画效果（P1）

**描述**：应用需要有好看的动画效果，动画风格遵循"水墨物理学"原则（见 DESIGN.md）。

**边界**：
- 范围内：页面/组件进入退出动画（墨水扩散/褪去）
- 范围内：交互反馈动画（悬停、点击、拖拽）
- 范围内：加载状态动画（墨水晕染）
- 范围内：模板切换过渡动画
- 不在范围内：影响可用性的动画（不可为了好看牺牲操作效率）

**验收标准**：
- 核心交互（页面导航、模板切换、AI 优化等待）有动画过渡
- 动画时长遵循 DESIGN.md 中的规范（进入 300-500ms，退出 150-250ms）
- 提供减少动画的无障碍选项（`prefers-reduced-motion`）
- 动画不导致界面卡顿（60fps 基线）

## 5. 用户体验

### 5.1 降低认知负担（P0）

**描述**：这是 intent.md 中明确提出的 UX 原则。产品应让用户专注于简历内容本身，而非工具操作。

**边界**：
- 范围内：简洁的导航结构，核心流程（编辑 → 优化 → 导出）清晰可见
- 范围内：AI 功能以预设选项呈现，不暴露底层复杂性
- 范围内：合理的默认值（默认模板、默认简历结构）
- 已裁决：冷启动通过 AI 生成模式解决（用户描述背景 → AI 生成初稿），无需示例数据预填
- 不在范围内：面向高级用户的深度定制面板（如自定义提示词、Typst 源码编辑）

**验收标准**：
- 新用户首次使用，从打开应用到产出第一份简历 PDF 的操作路径不超过 5 个主要步骤
- 所有核心功能（编辑、AI 优化、模板切换、导出）从主界面可在 2 次点击内到达
- 无需阅读帮助文档即可完成基本操作

### 5.2 响应式设计（P1）

**描述**：应用应在不同设备和屏幕尺寸下正常使用。

**边界**：
- 范围内：桌面端浏览器完整功能
- 已裁决：仅桌面端（≥1024px），不做移动端专门适配

**验收标准**：
- 桌面端（≥1024px 宽度）所有功能正常运作
- 古风设计元素在各分辨率下视觉一致

### Q14: 富文本编辑的范围和优先级 [F04]
**背景**：requirements.md 1.1 明确写了"范围内：富文本编辑（加粗、斜体、列表等基础格式）"，但当前所有 description 字段使用纯文本 textarea。富文本编辑的实现复杂度高（需要 editor 库、Typst 模板支持 markup 解析、AI 序列化/反序列化需处理格式标记）。
