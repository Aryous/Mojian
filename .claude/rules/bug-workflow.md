# Bug 工作流

Bug 报告不是实现指令，而是一条待验证的事实。

## 1. 收集事实

至少记录以下内容：
- 复现步骤
- 实际结果
- 预期结果
- 影响范围
- 证据（报错、截图、日志、录屏、请求体）

如果信息不足，先补最小复现信息，不要立刻猜根因。

## 2. 先复现，再分类

Bug 必须先被复现为下列一种：
- **实现缺陷**：需求存在，代码行为错误
- **需求缺口**：用户合理期望未被需求或走查覆盖
- **需求与实现不一致**：需求写了，代码实现偏了
- **待裁决问题**：多种修法会改变产品或技术决策

## 3. 落库规则

- 实现缺陷：写入或扩展对应 `docs/exec-plans/active/*.md`
- 需求缺口 / 需求与实现不一致：在 `requirements.md` 的走查发现中补 `F` 条目
- 待裁决问题：在对应文档追加 `Q`，等待人类选择

口头结论不算落库。

## 4. 进入修复前的最小产物

开始修复前，至少要有以下产物之一：
- 一个失败测试
- 一段稳定复现脚本
- 一条可重复观察到的 UI / 日志 / 请求证据

没有可重复产物，就没有稳定闭环。

## 5. 修复顺序

1. 复现
2. 根因分析
3. 计划落库
4. 失败测试或复现证据
5. 修复代码
6. 验证修复
7. 刷新 `.claude/STATE.yaml`，并更新 exec-plan、必要规则；如需补人工备注，再写 `.claude/PIPELINE.md`

## 6. 验证要求

至少完成：

```bash
bash .claude/scripts/trace.sh
npm run lint
npx tsc -b --noEmit
npm test
```

如果是用户可见 Bug，还应补一条运行时验证：
- UI 路径重走
- 请求体核对
- 日志 / 输出比对

## 7. 关闭条件

一个 Bug 只有同时满足以下条件才能算关闭：
- 根因已知
- 修复已落地
- 行为已验证
- 文档 / 计划 / 缺口状态已更新
- 若暴露出 Harness 缺口，已编码为规则、脚本或测试
