# 主控查询路由表

当用户在问问题而非要求行动时，主控不走启动漏斗，直接按本表定位信息源。

---

## 状态查询

| 用户意图 | 动作 |
|---|---|
| "当前在哪个阶段" | 读 `.claude/STATE.yaml` |
| "哪些需求没完成" | 读 `requirements.md` → 读 `TRACE_INDEX.yaml` |
| "某个文档覆盖了什么" | 读对应的 `*.trace.yaml` |
| "为什么不能提交" | 跑 `closeout.sh` |
| "为什么被豁免放行" | 读 `docs/exemptions/*.md` |
| "当前有什么阻塞" | 读 `STATE.yaml → blockers` |

## 行动决策

| 用户意图 | 动作 |
|---|---|
| "接下来该干什么" | 读 `STATE.yaml → recommended_next`，按工作流分支 |
| "这个 Bug 怎么修" | 走 `bug-workflow.md` 流程（rules/ 已自动加载） |
| "我要加一个新功能" | 从 intent 开始走完整管线 |
| "帮我直接改代码" | 确认用户知道代价，记录上下文，再执行 |

## 裁决与回注

| 用户意图 | 动作 |
|---|---|
| "我选方案 X" | 按 `protocols.md` 裁决回注协议写回对应文档 |
| "这个问题不做了" | 在对应文档标注 `excluded`，更新 trace |
| "F条目丢弃" | 更新走查表状态 + 刷新 STATE |

## 系统维护

| 用户意图 | 动作 |
|---|---|
| "溯源覆盖率是多少" | 跑 `trace.sh`（注意：这是代码标注覆盖率，不是需求完成度） |
| "文档状态对不对" | 跑 `doc-lint.sh` |
| "帮我创建豁免" | 跑 `create-exemption.sh` |
| "Harness 自身有问题" | 诊断后改 `.claude/` 下的规则、脚本、协议 |

## 兜底规则

| 情况 | 动作 |
|---|---|
| 用户意图不匹配以上任何一条 | 先问清楚意图，不要猜 |
| 用户问的是概念/机制解释 | 先定位真相源文档，读完再回答，不从记忆回答 |
| trace.sh 报告与需求完成度混淆 | 代码标注覆盖率 ≠ 需求完成度，回答后者必须读 requirements.md |
