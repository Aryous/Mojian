# tech-decisions.md 文档结构契约

## Frontmatter

```yaml
---
status: draft | review | approved
author: tech-selection
date: YYYY-MM-DD
blocks: [design, feature]
open_questions: 0
---
```

- `approved` 时 `open_questions` 必须为 0
- 文档中仍有未决 Q 时，status 必须保持 `review`

## 章节结构

```
# 技术决策文档

---

## 决策 N: [决策标题]

**背景**：为什么需要做这个决策
**候选方案**：
- A: [方案描述]
- B: [方案描述]
- C: [方案描述]（可选）

**最终决策**：[选中的方案]
**选择理由**：[为什么选这个]
**未选原因**：
- [方案X]：[为什么没选]
**约束**：[使用限制、注意事项]

---

## 已裁决问题

| # | 问题 | 裁决 | 裁决人 | 日期 |
|---|---|---|---|---|

## 待人类裁决（如有）

### QX: [问题标题]
**背景**
**选项**（至少两个）
**影响**
**阻塞**
```

## 每个决策必须包含

1. **背景** — 为什么需要决策
2. **候选方案** — 至少 2 个
3. **最终决策** — 选了哪个
4. **选择理由** — 为什么选
5. **未选原因** — 每个未选方案的明确原因
6. **约束** — 使用限制

## 禁止事项

- 不得只推荐方案不记录未选原因
- 不得提出违反 ARCHITECTURE.md 依赖规则的方案
- 不得在没有决策记录的情况下建议修改 package.json
- 不得将 status 设为 approved（由人类审批）
